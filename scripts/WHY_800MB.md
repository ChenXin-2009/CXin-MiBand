# 为什么原始数据库有 800MB？

## 🔍 真相揭晓

### 原始数据库构成

```
总大小: 811.85 MB
├─ 字段数据: 226.62 MB (27.9%)
└─ 索引+元数据: 585.24 MB (72.1%) ⚠️ 这是主要原因！
```

**关键发现：索引占了 72.1% 的空间！**

## 📊 详细分解

### 字段数据（226.62 MB，27.9%）

| 字段 | 大小 | 占比 | 说明 |
|------|------|------|------|
| integers (id, collins, oxford, bnc, frq) | 64.90 MB | 8.0% | 5个整数字段 |
| word | 49.03 MB | 6.0% | 必需 |
| sw | 45.35 MB | 5.6% | 可删除 |
| translation | 40.43 MB | 5.0% | 必需（可压缩） |
| definition | 16.23 MB | 2.0% | 可删除 |
| exchange | 6.36 MB | 0.8% | 可删除 |
| phonetic | 3.88 MB | 0.5% | 必需 |
| 其他 | 0.43 MB | 0.1% | 可删除 |

### 索引+元数据（585.24 MB，72.1%）

**6 个索引，每个约 97.54 MB：**

1. `sqlite_autoindex_stardict_1` - 自动索引（id 主键）
2. `sqlite_autoindex_stardict_2` - 自动索引（word 唯一）
3. `stardict_1` - UNIQUE INDEX on id
4. `stardict_2` - UNIQUE INDEX on word
5. `stardict_3` - INDEX on (sw, word collate nocase)
6. `sd_1` - INDEX on (word collate nocase)

**每个索引约占 97.54 MB！**

## 💡 为什么索引这么大？

### 索引大小计算

```
340万词条 × 6个索引 = 2040万个索引条目
每个索引条目约 30 字节
总计：~585 MB
```

### B-Tree 索引结构

SQLite 使用 B-Tree 索引：
- 需要存储树节点
- 需要存储指针
- 需要存储排序信息
- `COLLATE NOCASE` 增加额外开销

## 🎯 优化效果分解

### 策略 1：只删除多余索引（保留所有词条）

```
保留 340万词，删除 5个索引，只保留 1个
预计大小: 324.15 MB
节省: 487.70 MB (60.1%)
```

**结论**：索引是最大的问题！

### 策略 2：只减少词条（保留所有字段和索引）

```
只保留 4.7万词，保留所有字段和索引
预计大小: 11.17 MB
节省: 800.68 MB (98.6%)
```

**结论**：词条数量也很关键！

### 策略 3：综合优化（推荐）

```
1. 减少词条: 340万 → 4.7万 (减少 98.6%)
2. 删除字段: 15个 → 5个 (减少 67%)
3. 压缩 translation: 限制 150 字符
4. 删除索引: 6个 → 1个 (减少 83%)

最终大小: 4.02 MB
压缩率: 99.5%
```

## ✅ 结论

### 为什么原始数据库有 800MB？

1. **索引太多**（6个）→ 占 585 MB（72%）
2. **词条太多**（340万）→ 基数大
3. **字段太多**（15个）→ 每词占用多
4. **没有压缩** → translation 可以更短

### 为什么优化后只需 4MB？

1. **只保留 1 个索引** → 节省 ~490 MB
2. **只保留 4.7万词** → 节省 ~800 MB（按比例）
3. **只保留 5 个字段** → 节省 ~150 MB
4. **压缩 translation** → 节省 ~20 MB

### 30MB 可以放多少词？

基于优化后的 90.09 字节/词：

```
30 MB ÷ 90.09 B = 349,171 词
```

**答案：34万+ 词！**

这已经是一个非常强大的词典了！
